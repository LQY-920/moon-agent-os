# moon-agent-os 平台愿景与子系统拆解

> 本文档是 moon-agent-os 的**顶层愿景设计**,不是一次性落地的 spec。
> 目的是锁定平台的核心定位、顶层架构、独立子系统清单、以及子系统之间的依赖与实现顺序。
> 每个子系统后续需要独立进入 brainstorming → writing-plans → 实现的完整循环。

**生成日期**: 2026-04-22
**参考材料**: `docs/refs/2026-04-22-ri-pao-article.md`(钉钉陈航关于"日抛型软件"的演讲)

---

## 一、核心理念

平台的核心概念是 **"日抛型软件"** —— 软件按需生成、用过即毁,但使用它所产生的**知识与记忆**持久留存并持续进化。

概念边界(防歧义):

- **日抛的是"软件形态"**:具体的 app / 页面 / 工作流实例是临时的,用完可被丢弃或重新生成。
- **持久的是"用户的记忆与意图"**:偏好、历史、上下文、经验沉淀在**用户级统一记忆中枢**里,跨所有产物共享。
- **"按日进化"的含义**:同一个意图第二次被表达时,平台基于记忆生成的产物质量应明显优于第一次。这是平台的核心差异化承诺。

这不是"每次都重头开始"的真·日抛,也不是"软件长期存在"的传统模式。是**软件日抛、记忆进化**。

---

## 二、关键决策(约束)

| 维度 | 决策 | 说明 |
|---|---|---|
| 目标用户 | 通用产品/平台(给别人用) | 必须包含账户、数据隔离、分发等完整平台能力 |
| 日抛语义 | 按日进化 | 软件瞬时、记忆持久 |
| 技术底座 | 构建在现有生态之上(编排层) | 模型/MCP/Claude Code 等都作为底层依赖,平台做编排、记忆、治理、体验 |
| 多形态关系 | 意图优先 + 形态优先 双路径并存 | 用户既可以"说需求让平台选形态",也可以"直接指定形态" |
| 记忆模型 | 用户级统一记忆中枢 | 单一归属主体,跨产物共享,简化权限与一致性 |
| 分发模型 | 有公共市场(类 GPTs Store) | 创作者生态作为长期愿景的一部分 |

## 三、顶层架构:四层洋葱

```
         ┌──────────────────────────────────────┐
         │     L5 分发层 Marketplace & Share    │  市场、fork、版本、治理
         ├──────────────────────────────────────┤
         │     L4 形态层 Surfaces                │  app/网页/小程序/MCP/
         │                                      │  Skill/工作流/Agent
         ├──────────────────────────────────────┤
         │     L3 生成 + 运行引擎 Forge & Runtime│  意图→产物、调度、沙箱、
         │                                      │  迭代循环(按日进化机器)
         ├──────────────────────────────────────┤
         │     L2 记忆中枢 Memory Core           │  用户级统一记忆
         ├──────────────────────────────────────┤
         │     L1 治理底座 Governance            │  账户、权限、审计、计费
         └──────────────────────────────────────┘
```

**切分轴**: 抽象层次。下层支撑上层,每层可独立迭代。

**选择理由**:

1. 与"按日进化"的记忆语义天然契合 —— 记忆是底座,上层可碎、可重生,底下不动。
2. 与"编排层"技术策略一致 —— 洋葱层次映射"外部生态 → 平台编排 → 用户形态"。
3. 依赖关系最容易排实现顺序,从底向上不会中途推倒重来。

## 四、独立子系统清单

共 **5 层 · 28 个子系统**(L1: 4 · L2: 5 · L3: 6 · L4: 8 · L5: 5)。每个子系统都具备独立 brainstorm 的条件(清晰输入/输出、能单独立项、失败不拖垮邻居)。

### L1 · 治理底座 (Governance)

任何多用户平台都跑不掉的地基。不是业务亮点,但是"没有就开不了张"。

| ID | 子系统 | 核心职责 |
|---|---|---|
| S1.1 | 账户与身份 (Identity) | 注册/登录/多设备会话;预留 OAuth、SSO 接口 |
| S1.2 | 权限与策略 (Policy) | 产物/记忆/市场的可见性、可操作性规则引擎 |
| S1.3 | 审计与操作日志 (Audit) | 记录谁在什么时候做了什么;是"听记"的基础设施 |
| S1.4 | 计费与配额 (Billing & Usage) | 用量采集、配额、成本计量;**展示归产品侧**(见跨形态约束) |

### L2 · 记忆中枢 (Memory Core)

整个平台的"灵魂"。"按日进化"中的"进化"发生在这一层,而不是产物层。用户级统一中枢。

| ID | 子系统 | 核心职责 |
|---|---|---|
| S2.1 | 记忆存储 (Memory Store) | 结构化 + 非结构化(会话、文档、产物快照、交互轨迹)的统一仓 |
| S2.2 | 采集管道 (Ingest) | 从各形态产物、用户上传、外部数据源持续抽取;对应"听记" |
| S2.3 | 理解与索引 (Understand & Index) | 结构化、向量化、打标签、去重、冲突检测 |
| S2.4 | 回忆接口 (Recall API) | 给 Forge/Runtime/Agent 用的统一检索与上下文构造 API |
| S2.5 | 记忆治理 (Memory Governance) | 用户可查、可改、可删、可导出;隐私分级;遗忘策略 |

### L3 · 生成 + 运行引擎 (Forge & Runtime)

把"意图 → 产物"和"产物 → 跑起来"两件事做成工业化流水线。

| ID | 子系统 | 核心职责 |
|---|---|---|
| S3.1 | 意图捕获 (Intent) | 多轮澄清、意图结构化、形态推荐 |
| S3.2 | 产物模型 (Artifact Model) | 描述一个产物的统一数据结构(不管最终是 app 还是 MCP);版本、血缘、元数据。**平台契约中心** |
| S3.3 | 生成流水线 (Forge Pipeline) | 编排模型调用、模板、代码生成、资源装配 |
| S3.4 | 运行时与沙箱 (Runtime & Sandbox) | 产物执行环境;web 容器、应用壳、工作流引擎等的统一接入 |
| S3.5 | 迭代循环 (Iteration Loop) | "按日进化"的机器;收集产物运行反馈 → 写回记忆 → 影响下一次生成。**平台差异化开关** |
| S3.6 | 能力目录 (Capability Registry) | MCP/skill/外部 API 的注册、发现、组合;Forge 产物用什么,都从这里选 |

### L4 · 形态层 (Surfaces)

七种形态在这里各自落地,每个是独立子系统,彼此可并行演进,统一接 Artifact Model。

| ID | 子系统 | 核心职责 |
|---|---|---|
| S4.1 | 网页形态 (Web) | 浏览器内即时生成的 web 应用壳 |
| S4.2 | App 形态 (App) | 打包/Hybrid 壳 |
| S4.3 | 小程序形态 (Mini-program) | 微信/支付宝等生态适配层 |
| S4.4 | MCP 形态 (MCP) | 产物作为 MCP server 暴露给外部 AI |
| S4.5 | Skill 形态 (Skill) | Claude Code / 类似 harness 可加载的技能包 |
| S4.6 | 工作流形态 (Workflow) | 节点化自动化流程 |
| S4.7 | Agent 形态 (Agent) | 长驻自主智能体 |
| S4.8 | 形态网关 (Surface Gateway) | 把同一个 Artifact 分发/投影到多形态的路由器;**支撑"意图优先"路径** |

**跨形态约束**: 每个形态需要在其运行界面中呈现该产物的**成本与用量**(token/API 调用/存储/时长)。这是"成本/用量展示归产品"决策的落地要求,不独立成一个子系统。

**关于"轻应用"**: 讨论中曾出现,最终决定不作为独立形态纳入。需要"平台内内嵌运行"的能力由 S4.1 网页形态的 in-platform 变体承担即可。

### L5 · 分发层 (Marketplace)

让创作者的产物/能力流动起来,类 GPTs Store 但面向 7 种形态。

| ID | 子系统 | 核心职责 |
|---|---|---|
| S5.1 | 市场目录 (Catalog) | 发现、搜索、分类、评分 |
| S5.2 | 发布流水线 (Publish Pipeline) | 从 Artifact 到可分发物的打包、审核、版本化 |
| S5.3 | 安装与派生 (Install & Fork) | 其他用户安装/fork;**记忆在派生时如何处理是关键设计点**(留到子系统 brainstorm 再决) |
| S5.4 | 内容治理 (Content Moderation) | 质量、合规、敏感内容过滤 |
| S5.5 | 创作者激励 (Creator,远期) | 数据看板、排行;变现/分成预留位 |

## 五、依赖关系(DAG)

```
                                 ┌──────────────────────┐
                                 │   L5 分发层          │
                                 └──────────┬───────────┘
                                            │ 依赖产物模型 + 记忆分片
                                            ▼
              ┌─────────────────────────────────────────────────┐
              │                L4 形态层                         │
              │   S4.1~S4.7 各形态     S4.8 形态网关             │
              └──────┬──────────────────────────┬───────────────┘
                     │ 消费 Artifact             │ 向形态路由
                     ▼                          │
         ┌───────────────────────────────────┐  │
         │      L3 生成 + 运行引擎            │◀─┘
         │   S3.1 意图  S3.2 产物模型        │
         │   S3.3 流水线 S3.4 运行时         │
         │   S3.5 迭代循环 S3.6 能力目录     │
         └─────┬──────────────────┬──────────┘
               │                  │
               │ 读写记忆         │ 装载能力
               ▼                  │
     ┌─────────────────────┐     │
     │   L2 记忆中枢        │    │
     │   S2.1~S2.5         │    │
     └─────────┬───────────┘    │
               │                  │
               ▼                  ▼
         ┌───────────────────────────────┐
         │      L1 治理底座              │
         │   S1.1~S1.4                  │
         └───────────────────────────────┘
```

**关键边含义**:

- 所有层最终依赖 L1(身份/权限是前置)
- L3 是核心枢纽 —— 产物模型 S3.2 是整个平台的契约中心
- L2 和 L3 之间是**双向依赖**:生成时读记忆,运行后写回记忆(即 S3.5 迭代循环)
- L5 只依赖"已经稳定的产物和记忆",所以位于最上层

## 六、实现顺序(6 个里程碑)

一个人开发不强求严格线性,但每个里程碑都是**能停下来验证的自洽切片**。

### M0 · 地基(L1 最小子集)

- **子系统**: S1.1 身份 + S1.2 策略(最简版)
- **产出**: 用户能注册/登录,有"我的数据属于我"的基础隔离
- **为何第一个**: 后面所有东西都要挂在用户身上,不做就全是临时变量
- **规模**: 小

### M1 · 记忆骨架(L2 最小可用)

- **子系统**: S2.1 存储 + S2.4 回忆 API(最小版:文本会话 + 简单检索)
- **产出**: 用户级统一记忆跑起来,能存能取,但尚无"理解"能力
- **为何在形态之前**: 没有记忆,所有产物都是一次性的,不符合"按日进化"定位
- **规模**: 中

### M2 · 第一个产物走通(L3 骨干 + L4 第一形态)· MVP

- **子系统**:
  - L3: S3.1 意图 + S3.2 产物模型 + S3.3 流水线 + S3.4 运行时(仅支持一种) + S3.6 能力目录(最小)
  - L4: **只做 S4.1 网页形态**
- **产出**: 第一条完整流水线 —— 用户说意图 → 平台生成可运行网页 → 用完关闭
- **为何选网页作第一形态**: 零分发成本、生成链路最成熟、UI 即时可见、排查容易
- **规模**: 大(整个平台最重的里程碑)

### M3 · 进化闭环(L3 核心特色 + L2 加深)

- **子系统**:
  - S3.5 迭代循环(反馈写回记忆,下次生成参考上次)
  - S2.2 采集 + S2.3 理解索引(记忆不再是纯文本仓)
  - S2.5 记忆治理(用户可查可删)
- **产出**: **"日抛型软件 + 按日进化"真正兑现的时刻**。同一意图第二次比第一次明显更好
- **为何独立**: M2 跑通的是一次性生成,没有"进化"。这是平台特色从演示变为事实的分水岭,风险最高、最需要打磨
- **规模**: 大

### M4 · 形态扩展 + 双路径并存(L4 铺开 + S4.8)

按优先级铺设剩余形态:
- **先做**: S4.5 Skill、S4.4 MCP(技术门槛低、自己能用、反向丰富平台能力)
- **再做**: S4.7 Agent、S4.6 工作流(依赖成熟的运行时)
- **最后**: S4.2 App、S4.3 小程序(生态适配最重)
- **同步**: S4.8 形态网关 —— 实现"意图优先"路径(一个意图多端投影)

- **产出**: 平台从"只能生成网页"扩展到覆盖多形态;双路径(意图优先 / 形态优先)都可用
- **为何拆成一个里程碑**: 形态可并行,实际铺设顺序可依自身使用需求动态调整。Skill 和 MCP 优先是因为它们是**放大平台自身能力的杠杆**
- **规模**: 每个形态小-中;整体视铺设范围而定

### M5 · 分发 + 治理补齐(L5 全层 + L1 补齐)

- **子系统**:
  - S5.1~S5.4 市场核心(目录、发布、安装/fork、内容治理)
  - S5.5 创作者激励 先只做数据看板
  - S1.3 审计(有市场后合规刚性才出现)
  - S1.4 计费与用量(按产品侧定义,集成到各形态运行面板)
- **产出**: 平台从单用户工具升级为有网络效应的市场;愿景完整落地
- **为何最后**: 分发的价值来自被分发的东西足够好。M2/M3/M4 产出足够优质后,市场才有意义。过早做市场等于摆空货架
- **规模**: 中

## 七、战略性说明

### 贯穿多里程碑持续进化的子系统

- **S2.x 记忆**: M1 起步 → M3 加深 → M5 适配分发
- **S3.2 产物模型**: M2 立骨架 → M4 每加一种形态都会逼它演化
- **S1.x 治理**: M0 最小版起步,每个里程碑都在补

### 主动延后到对应子系统 brainstorm 时再决的设计点

- 各形态的具体技术栈(M2 / M4)
- 记忆的具体存储选型与向量化方案(M1)
- 沙箱/运行时的隔离级别与技术选型(M2)
- 市场的信任与评分机制(M5)
- 安装/fork 时记忆的迁移策略(S5.3)
- 成本/用量在各形态运行面板中的具体呈现(M4 / M5)

### 风险与验证节奏

- **最大风险在 M3**: "进化"做不出来,平台特色就塌了。建议 M2 一跑通就立刻做 M3 的最小验证(哪怕手工触发),先证明路径走得通,再铺形态。
- **M4 铺设顺序可调**: 按自身最想用的形态先做,不必严格按文档列出的顺序。

### 暂不纳入的讨论点(保留但不立项)

- 独立的开发者/扩展 SDK(给平台写插件)— 近期不独立,按需嵌入各子系统
- 独立的通知/触发器子系统 — 归入 S4.6 工作流 和 S4.7 Agent
- 独立的多端同步/离线子系统 — 归入 S2.1 存储 和 各 L4 形态

## 八、后续工作指引

本文档**不是实现计划**。每个子系统需要独立走一次完整的设计与实现循环:

```
  本文档(顶层愿景)
       │
       ▼
  选定一个子系统(推荐按 M0 → M5 顺序)
       │
       ▼
  brainstorming  ──→  该子系统的 design.md
       │
       ▼
  writing-plans  ──→  该子系统的 plan.md
       │
       ▼
  实现
```

推荐的下一次 brainstorm 起点: **M0 里的 S1.1 账户与身份** 或 直接跳到 **M2 里的 S3.2 产物模型**(如果想先把"平台契约中心"的形状讨论清楚)。
